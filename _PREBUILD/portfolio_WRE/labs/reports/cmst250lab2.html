Objectives
This was a lab completed in two parts in two sessions. The first part (and session) of the lab was centered around learning to use Unix and discovering how computers were operated prior to the development of graphical user interfaces (GUIs) that are commonplace (and thus we take for granted) today.  The second part of the lab involved assembly of a Raspberry Pi computer and installment of the Raspbian operating system on that computer, followed by learning how to actually use it. This was done in a ‘wing it, figure it out as you go’ fashion not unlike how computer systems technicians would learn to use new operating systems and software, and as such, no handout was given for this assignment.

Notes, Observations, and Questions
Part One - Computer Hardware and Learning Unix
The lab began with an examination of some older computer hardware to observe how computer technology has evolved over the years and to find the rudimentary components that make up a computer: the central processing unit, the memory drive (a.k.a. storage), and the input and output ports.  Then, I (along with each other student) was given a CanaKit Raspberry Pi 3 computer kit, instructed to examine the motherboard in search of the same components. For the remainder of this part of the lab, the task was to attempt to connect to the Professor's Raspberry Pi through Secure Shell, or SSH.  Initially, the plan was to do so through the Windows 10 computers present in the laboratory via a terminal app called PuTTY.  However, upon a discovery that the Raspberry Pi was on an entirely different network from the lab modules, everybody switched to their laptops to connect wirelessly to the correct network. I would have to take a slightly different route to get there because my laptop is a Chromebook.  Rather than PuTTY, Chrome OS uses a developer shell applet called crosh [Fig. 1], accessed by the keyboard shortcut Ctrl + Alt + T.  However, as I soon found out, crosh does not have support for the 'ssh' command.  To do that on Chrome, I needed to download the official extension for crosh called Secure Shell [Fig. 2], version 0.8.44.1.  Opening the extension brought up a menu similar to that of PuTTY.  From there, I attempted to connect with the provided IP address and port, which was denied.  The Professor cited all instances of difficulty connecting as Unix's security measures. As I was the only person who could not successfully connect, the reason for which I'm guessing is Chrome OS itself, I was temporarily given an iPad known to be able to connect (which I successfully did) so that I could continue with the lab.  The terminal app on Apple devices is Terminus. By using commands such as 'cd' (change the current directory you are in), 'ls' (list objects in the current directory), and 'touch' (create a file in the current directory, I learned to navigate a directory without a GUI as a visual aid.
Part Two - Building a Raspberry Pi
One week later, for the latter part of the lab, the goal was to assemble the Raspberry Pi 3 kits and install the Raspbian operating system to create and use functional Linux-based computers.  To do this, I followed the instructions for assembly in the case via the provided guide but had two small parts left over.  From asking, I learned that those objects were heat sinks to be placed over the two chips on the top side of the motherboard [Fig. 3]. Next, I had to download the package that contained the OS itself and load it onto the SD card.  The website presented two options of this package (entitled NOOBS): a regular version and a Lite version.  I selected the regular version for download.  In order to transfer it to the MicroSD card, I would have to format it first, as I found out from examining the README file within.  Another download later (SD Card Formatter for Windows), I was ready to hook up the card reader and the card itself to the laboratory module, format the card, extract and copy the NOOBS file into the card, and finally disconnect from the lab computer and insert the MicroSD-turned-memory-card into the Raspberry Pi 3.  There was a slot on the end where such an object would fit.
With the computer complete and loaded, I now only had to get it booted up.  I couldn't disconnect anything from the laboratory computers, as there were padlocks keeping their respective hardware in place (and of course I didn't have a key to gain access).  In order to actually initiate a startup, I had to hook up four the necessary pieces of equipment to create a usable GUI-based computer (see [Fig. 4] for a recreation).  Because of the shortage of available components, I was granted the projector, keyboard, power supply, and mouse that the Professor had used to demonstrate setting up the computer (thank you).  While he was scouring the technology center for parts so other students could set theirs up as well, I managed to boot mine up to the 'select an OS to install' prompt.  As per instructions, I chose the Raspbian OS.  No shorter than twenty minutes later, I finally got to a desktop [Fig. 5], set the timezone, set a password, and hooked my Raspberry Pi 3 up to a WiFi network.

Conclusions
This lab taught me quite a bit more than it intended to.  In the first part of the lab, aside from learning how to connect to a network and navigate a computer without a GUI, I discovered that a difference in operating systems, if great enough, can pose as a 'language barrier' of sorts that can prevent any forms of connection.  In the second portion of the lab, I learned how to assemble a computer's hardware and install an operating system over the course of fifteen minutes plus waiting time for everything to download, but I also truly experienced the 'trials and tribulations' learning style that most people in the computer field have to get used to when exploring a foreign operating system.  Yes, the labs didn't exactly lead to a definitely planned result on my end, but in the computer field that's just the nature of the beast.
